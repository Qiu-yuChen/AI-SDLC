# 员工临时车辆预约系统产品规格说明书

---

## 第一章：系统概述

### 1.1 项目背景  
随着公司业务的快速发展，员工日常通勤需求增加，部分员工在紧急或临时情况下需要使用公司车辆。为规范车辆使用流程、提高资源利用率、避免无序使用，特开发“员工临时车辆预约系统”。

### 1.2 系统目标  
- 实现员工临时用车的在线申请、审批、调度与记录  
- 提高车辆使用透明度，减少人为干预和资源浪费  
- 支持多部门、多岗位员工的灵活用车需求  
- 提供移动端支持，提升用户体验  

### 1.3 系统名称  
员工临时车辆预约系统（Temporary Vehicle Booking System, TVBS）

### 1.4 系统范围  
- 员工申请临时用车  
- 部门负责人/管理员审批  
- 车辆调度与状态管理  
- 用车记录与统计分析  
- 移动端支持（App/微信小程序）  

### 1.5 系统用户角色  
| 角色 | 权限说明 |
|------|----------|
| 员工 | 提交用车申请、查看审批状态、查看已用车记录 |
| 部门负责人 | 审批申请、查看本部门用车情况 |
| 行政/车辆管理员 | 管理车辆信息、审批权限设置、查看全局用车数据 |
| 系统管理员 | 系统配置、用户管理、日志审计 |

## 第二章：功能需求

### 2.1 员工用车申请  
- 员工可在线填写用车申请表单，包括：  
  - 用车人姓名、工号、所属部门  
  - 用车时间（起止时间，支持日期+时间）  
  - 用车事由（可选）  
  - 用车地点（起点、终点）  
  - 用车类型（如：出差、会议、紧急事务）  
- 支持上传附件（如：会议通知、紧急情况说明）  
- 提交后自动进入审批流程  

### 2.2 审批流程  
- 默认审批路径：员工 → 部门负责人 → 行政/车辆管理员（可配置）  
- 支持多级审批，可设置审批节点的审批时限（如：24小时内必须审批）  
- 审批状态实时更新（待审批、已通过、已拒绝、已超时）  
- 审批意见可填写，拒绝时可附说明  

### 2.3 车辆管理  
- 管理员可维护车辆信息：  
  - 车牌号、车型、所属部门、状态（可用/维修/停用）  
  - 车辆位置（可选：GPS定位）  
- 支持车辆状态实时更新，避免重复调度  

### 2.4 调度与用车分配  
- 系统根据申请时间、车辆可用性、距离等因素自动推荐可用车辆  
- 支持手动指定车辆（如：特定员工指定某车）  
- 用车成功后，系统自动更新车辆状态为“已使用”  
- 用车结束后，系统自动更新为“空闲”  

### 2.5 用车记录与统计  
- 每次用车生成记录，包含：  
  - 申请时间、用车时间、用车事由、用车人、车辆信息  
- 支持按时间、部门、用车类型等维度进行统计分析  
- 生成月度/季度用车报表（PDF/Excel导出）  

### 2.6 移动端支持  
- 支持微信小程序或App端使用  
- 支持扫码登录、消息推送（审批通知、用车提醒）  
- 支持离线缓存，网络恢复后自动同步  

### 2.7 权限与安全  
- 用户登录需通过公司账号（如OA系统集成）  
- 所有操作记录可追溯（操作人、时间、内容）  
- 敏感操作（如拒绝申请、修改车辆状态）需二次确认

## 第三章：非功能需求

### 3.1 性能要求  
- 支持并发用户数 ≥ 500  
- 页面加载时间 ≤ 2秒（网络正常）  
- 申请提交响应时间 ≤ 1秒  

### 3.2 可用性  
- 系统全年可用率 ≥ 99.9%  
- 支持7×24小时运行  

### 3.3 可扩展性  
- 支持未来扩展至跨部门、跨区域用车  
- 支持与OA系统、ERP系统对接（如：自动同步员工信息）  

### 3.4 安全性  
- 数据传输采用HTTPS加密  
- 用户密码加密存储（SHA-256或更高级）  
- 定期备份，支持灾难恢复（每日增量备份）  

### 3.5 兼容性  
- 浏览器兼容：Chrome、Edge、Firefox（最新版本）  
- 移动端兼容：iOS 12+，Android 8.0+

## 第四章：数据模型

### 4.1 核心数据表结构

| 表名 | 字段说明 |
|------|----------|
| `users` | id, name, employee_id, department_id, role, status, create_time |
| `vehicles` | id, license_plate, model, department_id, status, location, create_time |
| `booking_requests` | id, user_id, start_time, end_time, reason, origin, destination, type, status, create_time, approve_status, approve_time, approve_user_id |
| `approvals` | id, request_id, approver_id, status, comment, approve_time |
| `booking_logs` | id, request_id, action, operator, time, ip |

### 4.2 数据关系  
- `users` → `booking_requests`（1:N）  
- `vehicles` → `booking_requests`（1:N）  
- `booking_requests` → `approvals`（1:N）

## 第五章：系统架构

### 5.1 架构图（文字描述）  
```
用户端（Web/小程序）
     ↓
API网关（Nginx）
     ↓
后端服务（Spring Boot + MyBatis）
     ↓
数据库（MySQL 8.0）
     ↓
缓存（Redis，用于会话、热门数据）
     ↓
消息队列（RabbitMQ，用于异步通知）
     ↓
日志系统（ELK，用于审计与监控）
```

### 5.2 技术栈  
- 前端：Vue.js + Element UI（Web） / 微信小程序（WXML/WXSS）  
- 后端：Java 17 + Spring Boot + MyBatis  
- 数据库：MySQL 8.0（主从复制）  
- 缓存：Redis（缓存用户信息、审批状态）  
- 消息通知：微信模板消息、站内信  
- 部署：Docker + Nginx + Tomcat

## 第六章：部署与运维

### 6.1 部署环境  
- 服务器：Linux（CentOS 7/8）  
- 硬件要求：  
  - CPU：2核以上  
  - 内存：4GB以上  
  - 磁盘：10GB（系统）+ 50GB（数据）  
- 网络要求：公网IP，支持HTTPS访问  

### 6.2 部署方式  
- 支持本地部署（公司内网）  
- 支持云部署（阿里云/腾讯云）  

### 6.3 运维管理  
- 日志监控（通过ELK或Prometheus）  
- 定期备份（每日凌晨2点自动备份至异地）  
- 系统健康检查（自动检测服务状态）  
- 故障自动告警（邮件/钉钉）
