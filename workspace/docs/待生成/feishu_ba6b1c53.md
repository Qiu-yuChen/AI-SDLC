# 员工临时车辆预约系统产品规格说明书

---

## 第一章：系统概述

### 1.1 项目背景  
随着公司业务发展，员工日常通勤需求增加，为提升员工出行效率、优化车辆资源利用率，特开发“员工临时车辆预约系统”。该系统旨在为员工提供便捷、安全、透明的临时车辆使用服务，实现车辆资源的合理分配与管理。

### 1.2 系统目标  
- 实现员工临时用车的在线预约与审批流程  
- 提高车辆使用效率，减少车辆闲置  
- 增强员工用车体验，提升满意度  
- 支持灵活的用车时间、地点、车型选择  
- 保障用车安全，明确责任归属  

### 1.3 系统范围  
- 覆盖公司内部员工（经审批可使用）  
- 支持临时用车（非长期租赁）  
- 用车类型包括：私家车、公司公务车、共享车辆（可选）  
- 不包含车辆维修、保险、加油等服务  

### 1.4 目标用户  
- 员工（发起预约）  
- 行政/人事部门（审批与管理）  
- 车辆管理员（车辆状态维护）  
- 系统管理员（权限配置与数据监控）

## 第二章：功能需求

### 2.1 员工端功能

#### 2.1.1 登录与身份验证  
- 支持员工账号登录（手机号/邮箱+密码）  
- 支持企业微信/钉钉单点登录（SSO）  
- 登录失败次数限制与安全提醒  

#### 2.1.2 用车预约  
- 选择用车类型（私家车、公务车、共享车）  
- 选择用车时间（起止时间，支持小时级）  
- 选择用车地点（支持地图选择或输入）  
- 选择车型（根据车辆库存展示）  
- 填写用车目的（如：出差、接送、会议）  
- 提交预约申请，自动进入审批流程  

#### 2.1.3 预约状态查看  
- 实时查看预约状态（待审批、已审批、已用车、已取消）  
- 查看历史用车记录（按时间、类型、状态筛选）  
- 支持导出用车记录（Excel/CSV）  

#### 2.1.4 用车提醒  
- 提前1小时提醒审批人  
- 提前15分钟提醒用车人  
- 支持设置提醒方式（短信、站内信、APP通知）  

#### 2.1.5 取车与归还  
- 用车人可在线确认取车（需验证身份）  
- 归还车辆时自动记录时间，系统自动计算用车时长  
- 归还后自动更新车辆状态  

---

### 2.2 审批端功能

#### 2.2.1 审批流程配置  
- 支持多级审批流程（如：部门主管 → 行政部 → 总经理）  
- 支持自定义审批规则（如：用车时长超过3小时需额外审批）  

#### 2.2.2 审批操作  
- 查看待审批列表（按部门、时间、状态）  
- 批量审批（支持勾选多条）  
- 审批意见填写（可添加备注）  
- 审批结果自动同步至员工端  

#### 2.2.3 审批日志  
- 记录每次审批操作的时间、操作人、审批结果  
- 支持导出审批日志  

---

### 2.3 车辆管理端功能

#### 2.3.1 车辆信息管理  
- 添加/编辑车辆信息（车牌号、车型、颜色、所属部门、状态）  
- 支持上传车辆照片  
- 车辆状态管理（空闲、在用、维修、不可用）  

#### 2.3.2 车辆使用状态监控  
- 实时查看车辆当前状态（空闲/在用）  
- 支持按时间、地点、车型筛选车辆使用情况  
- 车辆使用时长统计（按日/周/月）  

#### 2.3.3 车辆维护提醒  
- 自动提醒车辆保养周期（如：每6个月保养）  
- 支持手动记录维修记录  

---

### 2.4 系统管理端功能

#### 2.4.1 用户管理  
- 添加/编辑员工账号（支持部门、职位、权限分配）  
- 设置员工权限（如：只读、可预约、可审批）  
- 支持批量导入员工信息（Excel模板）  

#### 2.4.2 角色与权限配置  
- 定义角色（员工、审批人、管理员）  
- 配置各角色可访问功能模块  
- 支持细粒度权限控制（如：仅查看历史记录）  

#### 2.4.3 系统设置  
- 配置用车规则（如：每日最多预约2次）  
- 设置审批流程模板（可复用）  
- 配置通知模板（短信/站内信）  
- 设置系统时间、时区、语言  

#### 2.4.4 数据统计与报表  
- 用车总量统计（按部门、时间）  
- 车辆使用率分析（图表展示）  
- 高频用车地点热力图  
- 用车成本分析（可选）

## 第三章：非功能需求

### 3.1 性能要求  
- 系统响应时间：页面加载 ≤ 2秒，操作响应 ≤ 1秒  
- 支持并发用户数 ≥ 1000  
- 预计日均预约量 ≥ 500 条  

### 3.2 安全性要求  
- 数据传输使用 HTTPS（TLS 1.2+）  
- 敏感信息加密存储（如：手机号、身份证）  
- 登录密码加密存储（使用 bcrypt）  
- 审计日志记录所有关键操作  
- 防止SQL注入、XSS、CSRF攻击  

### 3.3 可用性要求  
- 系统全年可用率 ≥ 99.9%  
- 支持7×24小时服务  
- 提供错误提示与操作引导  

### 3.4 兼容性要求  
- 支持主流浏览器：Chrome、Edge、Firefox、Safari（最新版本）  
- 支持移动端（H5或小程序）  
- 支持企业微信/钉钉集成  

### 3.5 可维护性  
- 代码结构清晰，模块化设计  
- 提供完整的日志系统与错误追踪  
- 支持热更新与灰度发布

## 第四章：数据模型

### 4.1 核心数据表

| 表名 | 字段说明 |
|------|----------|
| `users` | id, name, phone, email, department, role, status |
| `vehicles` | id, plate_number, model, color, status, department, create_time |
| `appointments` | id, user_id, vehicle_id, start_time, end_time, location, purpose, status, create_time, approve_time, remark |
| `approval_logs` | id, appointment_id, approver_id, status, remark, create_time |
| `vehicle_logs` | id, vehicle_id, action, description, create_time |
| `notifications` | id, user_id, type, content, send_time, status |

### 4.2 数据关系  
- 一个员工可预约多个用车记录  
- 一个用车记录关联一辆车  
- 一个用车记录经过多个审批节点  
- 车辆状态由用车记录动态更新

## 第五章：系统架构

### 5.1 架构图（文字描述）  
```
用户端（Web/APP/小程序）
     ↓
API网关（Nginx）
     ↓
微服务架构：
  - 用户服务（用户管理）
  - 车辆服务（车辆管理）
  - 预约服务（核心业务）
  - 审批服务（审批流程）
  - 通知服务（短信/站内信）
     ↓
数据库（MySQL主从+Redis缓存）
     ↓
消息队列（RabbitMQ/Kafka，用于异步通知）
     ↓
日志中心（ELK）
```

### 5.2 技术栈  
- 前端：Vue3 + Element Plus / Ant Design  
- 后端：Spring Boot（Java）或 Node.js  
- 数据库：MySQL 8.0（主从复制）  
- 缓存：Redis（缓存用户、车辆状态）  
- 消息队列：RabbitMQ（异步通知）  
- 文件存储：OSS（图片上传）  
- 部署：Docker + Kubernetes（K8s）  
- 安全：JWT + OAuth2.0

## 第六章：部署与运维

### 6.1 部署环境  
- 生产环境：Linux（CentOS 7/8）  
- 网络要求：内网访问，外网仅开放HTTPS端口（443）  
- 部署方式：容器化部署（Docker）  

### 6.2 运维流程  
- 日志监控：通过ELK系统实时查看错误日志  
- 性能监控：使用Prometheus + Grafana  
- 告警机制：当CPU > 80% 或错误率 > 1% 时自动通知运维  
- 备份策略：每日全量备份，每周增量备份，保留30天  

### 6.3 升级与维护  
- 支持灰度发布（先对部分用户开放）  
- 每月进行一次系统健康检查  
- 每季度进行一次数据迁移与优化
