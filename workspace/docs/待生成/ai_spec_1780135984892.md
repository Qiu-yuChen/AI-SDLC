# 车票管理系统产品规格说明书

## 1. 项目概述

### 1.1 背景
随着交通出行需求的增长，传统的人工售票方式效率低下、易出错，无法满足现代车站的运营需求。为提升售票、检票及票务管理效率，计划开发一套车票管理系统。

### 1.2 目标
- 实现车票的在线/线下销售、退票、改签功能。
- 提供实时余票查询、车次管理、座位分配。
- 支持多角色协同操作（管理员、售票员、乘客）。
- 系统稳定、安全、易用。

### 1.3 范围
- **包含**：车次管理、座位管理、售票、退票、改签、订单管理、用户管理、报表统计。
- **不包含**：支付网关对接（预留接口）、线下检票设备对接。

## 2. 功能需求

### 2.1 用户角色
| 角色   | 描述                                                         |
|--------|--------------------------------------------------------------|
| 乘客   | 查询车次、购票、退票、改签、查看订单。                       |
| 售票员 | 线下售票、退票、改签，管理当日车次。                         |
| 管理员 | 车次增删改查、座位配置、票价设置、用户管理、报表查看。       |

### 2.2 核心功能模块

#### 2.2.1 车次管理（管理员）
- 添加/编辑/删除车次（车次号、出发站、到达站、出发时间、到达时间、运行时长）。
- 配置车厢与座位（车厢号、座位类型、座位编号）。
- 设置票价（按座位类型、时间段浮动）。
- 车次状态（运营中、停运）。

#### 2.2.2 余票查询（所有角色）
- 按出发站、到达站、日期查询可用车次。
- 显示余票数量、票价、发车时间。
- 支持模糊搜索。

#### 2.2.3 售票（乘客/售票员）
- 选择车次、座位类型、乘客信息（姓名、身份证号）。
- 生成订单，锁定座位（超时未支付释放）。
- 支持多张票同时购买。
- 订单状态：待支付、已支付、已出票、已取消。

#### 2.2.4 退票（乘客/售票员）
- 选择已支付订单中的车票进行退票。
- 计算退票手续费（按规则：开车前>48h免手续费，24-48h收10%，<24h收20%）。
- 释放座位。

#### 2.2.5 改签（乘客/售票员）
- 选择原车票，改签至同一线路的其他车次（同日期或不同日期）。
- 差价多退少补。
- 仅允许改签一次。

#### 2.2.6 订单管理（乘客/售票员/管理员）
- 查看历史订单（按时间、状态筛选）。
- 管理员可查看所有订单。

#### 2.2.7 用户管理（管理员）
- 乘客注册/登录（手机号+验证码或密码）。
- 售票员账号由管理员创建。
- 权限管理。

#### 2.2.8 报表统计（管理员）
- 按日/周/月统计售票数量、收入、退票率。
- 热门车次排行。

### 2.3 业务流程（售票）
1. 用户查询车次。
2. 选择车次、座位、乘客信息。
3. 系统锁定座位，生成待支付订单。
4. 用户支付（预留支付接口）。
5. 支付成功后，订单状态变为已支付，座位确认。
6. 系统自动出票（生成电子票号）。
7. 用户可查看电子票。

## 3. 非功能性需求

### 3.1 性能
- 并发支持：高峰时段支持500用户同时查询购票。
- 响应时间：查询请求<2s，购票请求<3s。
- 座位锁定超时：15分钟自动释放。

### 3.2 安全
- 用户密码加密存储（bcrypt）。
- 接口身份验证（JWT Token）。
- 防止SQL注入、XSS攻击。
- 敏感操作日志记录。

### 3.3 可用性
- 系统可用性>99.5%。
- 支持主流浏览器（Chrome、Firefox、Edge）。
- 适配移动端（响应式设计）。

### 3.4 可维护性
- 前后端分离架构（Vue.js + Spring Boot / Node.js）。
- API文档自动生成（Swagger）。
- 代码版本管理（Git）。

| PUT    | /api/orders/{id}/cancel  | 取消订单（退票）         |
| PUT    | /api/orders/{id}/change  | 改签                     |
| GET    | /api/admin/trips         | 管理员查看所有车次       |
| POST   | /api/admin/trips         | 管理员添加车次           |
| PUT    | /api/admin/trips/{id}    | 管理员编辑车次           |
| DELETE | /api/admin/trips/{id}    | 管理员删除车次           |

## 6. 技术约束与假设

### 6.1 技术约束
- 前端：Vue.js 3 + Element Plus
- 后端：Java Spring Boot 2.x / Node.js Express
- 数据库：MySQL 8.0
- 缓存：Redis（用于座位锁定、会话管理）
- 部署：Docker容器化，Nginx反向代理
- API采用RESTful风格，JSON数据格式

### 6.2 假设
- 支付系统由第三方提供，本系统仅预留接口。
- 车次数据由管理员手动维护，不涉及实时交通数据对接。
- 用户通过手机号注册，身份证号仅用于购票验证。

## 7. 验收标准

| 编号 | 验收项                                       | 预期结果                     |
|------|----------------------------------------------|------------------------------|
| 1    | 用户可注册登录                               | 成功登录并获取Token          |
| 2    | 按条件查询车次，显示余票和票价               | 返回正确数据，响应<2s        |
| 3    | 成功购票，生成订单，座位被锁定               | 订单状态为pending，座位不可用 |
| 4    | 支付后订单状态变为已支付                     | 状态更新，座位确认           |
| 5    | 退票成功，释放座位，扣除手续费               | 订单状态refunded，座位可用   |
| 6    | 改签成功，差价计算正确                       | 新订单生成，原订单取消       |
| 7    | 管理员可添加、编辑、删除车次                 | 数据持久化，前端更新         |
| 8    | 管理员查看报表                               | 图表或表格数据正确           |
| 9    | 并发压力下系统稳定                           | 500用户同时操作无崩溃        |
| 10   | 未登录用户无法访问需鉴权接口                 | 返回401错误                  |
